# Native Infrastructure: Custom Modules, Renderers & Frameworks

## Overview
This is the frontier — going beyond using React Native's built-in systems to building your own custom C++ modules, rendering backends, and even entirely new frameworks on top of React Native's core. This is where you can eliminate the remaining performance gaps and build things that aren't possible in standard React Native.

---

## 1. The Architecture Stack (Full Picture)

```
┌─────────────────────────────────────────────────────────────────────┐
│  Level 5: Your Application (JS/React)                               │
├─────────────────────────────────────────────────────────────────────┤
│  Level 4: React Native Framework Layer (Navigation, Expo SDK, etc.) │
├─────────────────────────────────────────────────────────────────────┤
│  Level 3: React Native Core (Fabric, TurboModules, JSI, Yoga)       │
├─────────────────────────────────────────────────────────────────────┤
│  Level 2: JS Engine (Hermes/JSC)                                    │
├─────────────────────────────────────────────────────────────────────┤
│  Level 1: Native Platform (Android JVM/ART + NDK / iOS + UIKit)     │
└─────────────────────────────────────────────────────────────────────┘
```

When you build custom native infrastructure, you can insert your code at any level — typically Levels 2-3.

## 2. Custom TurboModule (C++ → JS)

The cleanest way to expose high-performance native functionality to React Native JS.

### Full Example: Custom FFT Module (Audio Processing)

```typescript
// specs/NativeFFTModule.ts
import type { TurboModule } from 'react-native';
import { TurboModuleRegistry } from 'react-native';

export interface Spec extends TurboModule {
  // Synchronous — JSI enables this
  computeFFT(samples: number[], windowSize: number): number[];
  // Returns magnitude spectrum
}

export default TurboModuleRegistry.getEnforcing<Spec>('NativeFFTModule');
```

```cpp
// cpp/NativeFFTModule.h
#pragma once
#include <NativeFFTModuleCxxSpec.h>  // generated by codegen
#include <fftw3.h>                   // FFTW library

namespace facebook::react {

class NativeFFTModule : public NativeFftModuleCxxSpec<NativeFFTModule> {
  fftw_plan plan_;
  
public:
  explicit NativeFFTModule(std::shared_ptr<CallInvoker> jsInvoker);
  
  std::vector<double> computeFFT(
    jsi::Runtime& rt,
    std::vector<double> samples,
    double windowSize
  );
};

} // namespace
```

```cpp
// cpp/NativeFFTModule.cpp
#include "NativeFFTModule.h"
#include <cmath>

namespace facebook::react {

std::vector<double> NativeFFTModule::computeFFT(
    jsi::Runtime& rt,
    std::vector<double> samples,
    double windowSize) {
  
  int n = (int)windowSize;
  
  // FFTW: compute FFT in-place
  auto* in  = fftw_alloc_real(n);
  auto* out = fftw_alloc_complex(n / 2 + 1);
  
  for (int i = 0; i < n; ++i) in[i] = samples[i];
  
  auto plan = fftw_plan_dft_r2c_1d(n, in, out, FFTW_ESTIMATE);
  fftw_execute(plan);
  
  // Compute magnitude spectrum
  std::vector<double> magnitudes;
  magnitudes.reserve(n / 2 + 1);
  for (int i = 0; i < n / 2 + 1; ++i) {
    double re = out[i][0], im = out[i][1];
    magnitudes.push_back(sqrt(re * re + im * im));
  }
  
  fftw_destroy_plan(plan);
  fftw_free(in);
  fftw_free(out);
  
  return magnitudes;
}

} // namespace
```

```js
// Usage in React Native JS (synchronous! no await needed)
import NativeFFTModule from './specs/NativeFFTModule';

const magnitudes = NativeFFTModule.computeFFT(audioSamples, 1024);
// Returns immediately — runs C++ on the calling thread (JS thread in this case)
// For large FFTs, dispatch to a C++ background thread and use a callback instead
```

## 3. JSI HostObject with Background Thread

For long-running native operations, run C++ on a background thread and post results back to JS:

```cpp
class AsyncProcessorModule : public jsi::HostObject {
  std::thread worker_;
  std::shared_ptr<CallInvoker> jsInvoker_;
  std::atomic<bool> running_{true};

public:
  explicit AsyncProcessorModule(std::shared_ptr<CallInvoker> jsInvoker)
    : jsInvoker_(jsInvoker) {}

  jsi::Value get(jsi::Runtime& rt, const jsi::PropNameID& name) override {
    if (name.utf8(rt) == "processAsync") {
      return jsi::Function::createFromHostFunction(
        rt, name, 2,
        [this](jsi::Runtime& rt, const jsi::Value& thisVal, const jsi::Value* args, size_t count) {
          
          // Copy data before spawning thread (rt and args are NOT safe to use off-thread)
          std::string data = args[0].getString(rt).utf8(rt);
          auto callback = std::make_shared<jsi::Function>(args[1].getObject(rt).asFunction(rt));
          
          // Spawn background thread
          std::thread([this, data, callback]() {
            // Run expensive work on background thread (NO JSI access here!)
            std::string result = doHeavyProcessing(data);
            
            // Post result back to JS thread via CallInvoker
            jsInvoker_->invokeAsync([callback, result](jsi::Runtime& rt) {
              // We're back on the JS thread — JSI access is safe again
              callback->call(rt, jsi::String::createFromUtf8(rt, result));
            });
          }).detach();
          
          return jsi::Value::undefined();
        }
      );
    }
    return jsi::Value::undefined();
  }
};
```

class CustomCanvasView(context: Context) : View(context) {
  private val paint = Paint()
  private var color: Int = Color.TRANSPARENT
  
  override fun onDraw(canvas: Canvas) {

    // Custom drawing using Android Canvas API

## 5. Custom React Reconciler (Completely Custom Renderer)

Using `react-reconciler`, you can build a renderer that doesn't create native views at all:

```js
// customRenderer.js
import ReactReconciler from 'react-reconciler';

const hostConfig = {
  // Called when React wants to create a new instance (element)
  createInstance(type, props, rootContainer) {
    // Create your own "node" type — could be a WebGL mesh, a physics body, etc.
    return { type, props, children: [], glMesh: createGLMesh(type, props) };
  },

  appendChildToContainer(container, child) {
    container.scene.add(child.glMesh);  // add to WebGL scene
  },

  removeChildFromContainer(container, child) {
    container.scene.remove(child.glMesh);
  },

  commitUpdate(instance, updatePayload, type, oldProps, newProps) {
    // Apply prop changes to your scene node
    applyPropsToMesh(instance.glMesh, newProps);
  },

  // Required: Fabric runs layout — return null to skip Yoga
  prepareForCommit() { return null; },
  resetAfterCommit() {},

  // Support text nodes?
  createTextInstance() {},

  // Enable concurrent mode features?
  supportsMutation: true,
  isPrimaryRenderer: false,  // true if this is the only renderer
  
  // ... 20+ more required methods (see react-reconciler docs)
};

const renderer = ReactReconciler(hostConfig);

// Create a React root targeting your custom container
export function render(element, container, callback) {
  const root = renderer.createContainer(container, 0, null, false, null, '', {}, null);
  renderer.updateContainer(element, root, null, callback);
}
```

This is exactly how **React Three Fiber** (3D React renderer using Three.js), **React Native Skia** (2D GPU renderer), and **React Native TVOS** work.

// Implement jsi::Runtime for your custom engine
class QuickJSRuntime : public jsi::Runtime {
public:
  // Create a JS value from a number
  jsi::Value makeNumber(double value) override {
    return JS_NewFloat64(ctx_, value); // QuickJS API
  }
  
  // Evaluate JS code
  jsi::Value evaluateJavaScript(const std::shared_ptr<const jsi::Buffer>& buffer,
                                  const std::string& sourceURL) override {
    JSValue result = JS_Eval(ctx_, 
      (const char*)buffer->data(), buffer->size(),
      sourceURL.c_str(), JS_EVAL_TYPE_GLOBAL);
    return convertJSValueToJSI(result);
  }
  
  // Create a host function
  jsi::Object createFunctionFromHostFunction(...) override { /* ... */ }
  
  // ... 60+ more required methods
};

// Register with React Native (Android)
class QuickJSExecutorFactory : public JSExecutorFactory {
public:
  std::unique_ptr<JSExecutor> createJSExecutor(
    std::shared_ptr<ExecutorDelegate> delegate,
    std::shared_ptr<MessageQueueThread> jsQueue) override {

## 7. Architectural Opportunities: What to Build

Here are the highest-impact areas for custom infrastructure:

### A) Zero-Copy Image Pipeline
```
Camera → Native YUV buffer (C++) → GPU texture (OpenGL/Vulkan) → screen
                                  ↑ No copies, no JS involvement per frame
```
Libraries like VisionCamera do this for camera preview. Extend it for real-time ML inference.

### B) Off-Main-Thread Layout Engine
Replace Yoga with a parallel, SIMD-optimized layout engine that distributes layout computation across CPU cores.

### C) Precompiled View Templates
Instead of running React reconciliation on every state change, precompile common UI patterns to native view blueprints that can be instantiated in 1-2ms instead of 10-20ms.

### D) Custom Animation Engine
Build a physics-based animation engine in C++ that runs on a dedicated render thread — no JS involvement, no Reanimated worklet overhead:
```cpp
class PhysicsAnimator {
  std::thread animationThread_;
  
  void runLoop() {
    while (running_) {
      auto now = std::chrono::high_resolution_clock::now();
      updatePhysics(dt);
      applyLayoutToNativeViews();  // direct UIThread call via CADisplayLink / Choreographer
      sleepUntilNextFrame();
    }
  }
};
```

### E) Persistent Render Thread (Vulkan/Metal)
Build a Vulkan (Android) or Metal (iOS) render thread that continuously composites React Native views + custom graphics without going through the platform compositor:
```cpp
// Android: Use ANativeWindow + Vulkan
ANativeWindow* window = ANativeWindow_fromSurface(env, jSurface);
VkSurfaceKHR surface;
vkCreateAndroidSurfaceKHR(instance, &createInfo, nullptr, &surface);
// Now render directly to the surface at any FPS you want
```
